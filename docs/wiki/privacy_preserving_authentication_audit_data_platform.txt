====== Privacy Preserving Authentication Audit Data Platform ======

===== Project Thesis =====

PramanaLedger is a privacy-preserving authentication audit data platform. The platform accepts authentication telemetry, validates the source contract, tokenizes sensitive identifiers, stores curated analytical records, preserves rejected records in quarantine, and writes batch-level audit evidence.

The project is designed as a banking-style data engineering system. The primary artifact is the governed data flow, not a decorative landing page.

===== Source Data Strategy =====

The live page uses deterministic synthetic authentication telemetry. This is deliberate. Real login-event datasets are usually sensitive, licensed, stale, or stripped of the exact operational fields needed to demonstrate privacy engineering.

The generated feed includes user identifiers, device types, raw device identifiers, IP-like addresses, locales, application versions, event timestamps, authentication outcomes, and risk bands. The raw fields are sufficient to demonstrate validation, tokenization, replay-safe identity, curated table loading, and operational metrics.

===== Why HMAC Tokenization =====

Plain hashes are weak for small input spaces. IPv4 addresses and common device identifiers can be dictionary-attacked. PramanaLedger uses secret-keyed HMAC-SHA256 so the token is stable for analytics but not useful without the secret.

This keeps routine reporting away from raw IP and device identifiers while still allowing the same token to be joined across repeated login events.

===== Event Identity and Replay Safety =====

The event identity is derived from a canonical source-event hash and a secret HMAC key. If the same event is replayed, it produces the same event_id. PostgreSQL enforces idempotency with ON CONFLICT DO NOTHING, so replay safety exists at the storage boundary.

===== Validation and Quarantine =====

The validation layer enforces required fields, device type domain checks, authentication outcome checks, risk-band checks, and parseable application versions. Invalid records are rejected rather than silently coerced.

===== Data Source Expansion Options =====

The safest near-term enhancement is deterministic synthetic telemetry plus stable public reference dimensions. Recommended candidates are ISO country and locale references, public autonomous-system reference tables, device taxonomy mappings, NIST privacy and security control references, and carefully reviewed public cyber-range logs.

The project should not scrape random public pages for login events. Authentication logs are privacy-sensitive by design, and a brittle public feed would weaken the credibility of the platform.
